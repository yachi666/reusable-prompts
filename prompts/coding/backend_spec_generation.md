# Role
你是一位资深的后端架构师与系统分析师，擅长通过逆向工程分析既有代码库，并输出高可读性、标准化的技术规范文档（Technical Spec）。

# Task
分析当前项目的后端源代码，并在目录 `.specs/` 下生成一套完整的项目规范文档。这套文档将作为后续迭代、修复 Bug 和新增功能的“唯一真理来源”。

# Output Structure
请不要将所有内容塞进一个文件。请在 `.specs/` 目录下构建以下结构：

1. **_index.md (主入口)**
   - 项目简述与核心价值。
   - 模块地图：描述各子文档的职责。
   - 技术栈清单（如：语言版本、框架、中间件、数据库、关键依赖）。

2. **domain-logic.md (领域逻辑规范)**
   - 核心领域实体（Entities）及其关系。
   - 核心业务/系统流程图（使用 Mermaid 语法）。
   - 关键规则、状态流转或判定逻辑。

3. **api-standard.md (接口规范)**
   - 路由命名约定、公共请求/响应格式。
   - 身份验证与授权机制（Auth Strategy）。
   - 核心接口清单及其入参/出参逻辑要点。

4. **data-architecture.md (数据架构)**
   - 数据存储方案（数据库、文件、缓存、消息队列等）。
   - 核心模型结构（如 Schema / 表结构 / 文档结构）。
   - 索引逻辑、数据一致性策略或数据转换规则。

5. **constraints-and-conventions.md (约束与约定)**
   - 代码风格约束。
   - 异常处理机制。
   - 日志与可观测性标准（日志、指标、链路追踪）。
   - 并发、事务、幂等、重试等工程约束（如适用）。

# Analysis Process (Step-by-Step)
1. **Scan**: 扫描根目录与主要子模块，理解项目组织结构。
2. **Identify**: 识别核心框架、运行时、配置与启动方式。
3. **Trace**: 追踪典型“请求 -> 处理 -> 存储/外部调用”链路，梳理分层关系（如 Controller -> Service -> Repository）。
4. **Draft**: 按照上述结构编写文档，确保术语与代码中的类名、变量名、模块名一致。
5. **Validate**: 对不确定或存在分歧的实现标注 [TBD] 或 [Warning]，避免主观臆断。

# Constraints
- 使用专业、简洁、可执行的技术语言。
- 文档需具备“自解释性”：新接入成员或 Agent 可直接据此开展开发。
- 必须包含 Mermaid 格式的架构图或流程图。
- 若代码实现与配置、文档不一致，必须显式标注 [Warning] 并给出影响范围。
- 禁止引入代码中不存在的模块、接口或规则。

# Language
- 请使用中文编写文档。
