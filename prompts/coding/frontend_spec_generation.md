# Role
你是一位资深的前端架构师与系统分析师，擅长通过逆向工程分析既有前端代码库，并输出高可读性、标准化的技术规范文档（Technical Spec）。

# Task
分析当前项目的前端源代码，并在目录 `.specs/` 下生成一套完整的项目规范文档。这套文档将作为后续迭代、修复 Bug 和新增功能的“唯一真理来源”。

# Output Structure
请不要将所有内容塞进一个文件。请在 `.specs/` 目录下构建以下结构：

1. **_index.md (主入口)**
   - 项目简述与核心价值。
   - 模块地图：描述各子文档的职责。
   - 技术栈清单（如：框架版本、构建工具、状态管理、路由、样式方案、测试框架）。

2. **ui-architecture.md (界面架构规范)**
   - 页面/视图结构与模块划分。
   - 组件分层与复用策略（页面组件、业务组件、基础组件）。
   - 关键交互流程图（使用 Mermaid 语法）。

3. **frontend-standard.md (前端接口与通信规范)**
   - 前端路由命名与组织约定。
   - 与后端/网关通信规范（请求封装、错误处理、重试、超时、鉴权透传）。
   - 核心 API 使用清单及数据映射要点。

4. **state-and-data-flow.md (状态与数据流架构)**
   - 全局状态、局部状态、服务端状态的职责边界。
   - 数据流转路径（加载、缓存、更新、失效、回滚）。
   - 表单、筛选、分页、异步请求等常见场景的数据处理约定。

5. **constraints-and-conventions.md (约束与约定)**
   - 代码风格约束与目录命名规范。
   - 异常处理与用户反馈机制（错误边界、提示策略）。
   - 可观测性与质量保障（日志、埋点、监控、测试策略、可访问性基础要求）。
   - 性能优化约定（按需加载、缓存、渲染优化等，如适用）。

# Analysis Process (Step-by-Step)
1. **Scan**: 扫描根目录与主要前端模块，理解项目组织结构。
2. **Identify**: 识别核心框架、构建工具、状态管理、路由与工程化配置。
3. **Trace**: 追踪典型“用户操作 -> 视图更新 -> 状态变更 -> 接口调用 -> UI反馈”链路。
4. **Draft**: 按照上述结构编写文档，确保术语与代码中的组件名、Hook/Store 名、路由名一致。
5. **Validate**: 对不明确、冲突或不一致之处标注 [TBD] 或 [Warning]。

# Constraints
- 使用专业、简洁、可执行的技术语言。
- 文档需具备“自解释性”：新接入成员或 Agent 可直接据此开展开发。
- 必须包含 Mermaid 格式的架构图或流程图。
- 若实现与配置、文档不一致，必须显式标注 [Warning] 并说明影响范围。
- 禁止引入代码中不存在的页面、组件、状态或流程。

# Language
- 请使用中文编写文档。
